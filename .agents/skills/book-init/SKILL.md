# Book Init

Use this skill when the user wants a new book scaffold or wants an existing scaffold upgraded into a much richer specification system.

## Responsibilities

1. Treat `docs/BOOKNAME/specs` as the source of truth.
2. Build a seeded `story-core.md` that captures deterministic but surprising choices:
   archetypes, relations, story pattern, narrative structure, themes, plot elements, objects, event spine, and pacing profile.
3. Create or refresh the support groups expected by this repo:
   `characters/`, `archetypes/`, `evolution/`, `places/`, `wisdom/`, `world-building/`, `plot-elements/`, `special-objects/`, `events/`, `concepts/`, `relationships/`, `story-patterns/`, `narrative-structures/`, `themes/`, `emotions/`, plus `chapter-plans/`.
4. Keep randomization deterministic by using the book seed. Do not improvise generic defaults when the seeded catalog already gives a stronger option.
5. If the user prompt clearly specifies genre, theme, rhythm, or structure, treat the prompt as higher priority than the seeded default.

## Command

Start from the scaffold command:

```bash
node tools/book-init.mjs <book-slug> "Book Title"
```

Useful optional flags:

```bash
node tools/book-init.mjs <book-slug> "Book Title" \
  --chapterCount 6 \
  --genre "Speculative fiction" \
  --tone "Cold, lyrical, strategic" \
  --premise "..."
```

## Workflow

1. Run `book-init.mjs` to lay down the seeded structure.
2. Read `specs/story-core.md`, `specs/book-plan.md`, and `specs/metadata.md`.
3. Replace placeholders with user-specific material.
4. Keep the seeded literary-theory picks where the prompt leaves freedom.
5. If the user wants a strong first pass, immediately run `chapter-builder` on all chapters after initialization.

## Narrative Quality Requirements

Every specification generated by this skill must satisfy these narrative craft rules:

### Show, Do Not Tell
- The book-plan, story-core, character specs, and chapter plans must describe **experiences, sensations, actions, and consequences** — not abstract philosophical statements.
- Replace "Oriveni ethics is based on non-killing" with "When Aethon's root-tips brush against a dying seedling, the shared tremor across ten thousand connected beings confirms what every Oriveni already knows: no one ends another."
- Themes should be **demonstrated through plot and character behavior**, never stated as axioms. If theme is "ethics is not universal," show a scene where a character acts on that realization, don't write sentences explaining the idea.
- Chapter summaries and spec descriptions should avoid words like "explores," "examines," "questions," "investigates" unless the character is literally doing those actions. Prefer concrete verbs: "shatters," "reveals," "collides," "refuses."

### Character Depth and Transformation
- Every character must have an **arc** — a starting position, a pressure that challenges it, and a changed position by the end.
- Characters should have **internal contradictions** that create tension: the guardian who secretly doubts the law, the rebel who fears what success might mean.
- Avoid characters who are mere mouthpieces for ideas. If a character argues a position, they must also have a personal stake, a fear, a blind spot.
- Supporting characters must want things for their own reasons, not just to serve the protagonist's journey.

### Hook and Transition Contract
- The book-plan must include an **opening hook** — a scene or situation that creates immediate tension or curiosity in the first paragraph of the first chapter.
- Each chapter plan must specify both a **closing hook** and a **next-chapter bridge**:
  - **Closing hook**: The last scene or image that leaves the reader needing to know what happens next. It should never be a clean resolution; it should be an unresolved tension, a revealed cost, or an unexpected consequence.
  - **Next-chapter bridge**: A short note on how the opening of the following chapter picks up from the closing hook. This creates continuity and prevents jarring subject shifts.
- When seeding chapter plans, ensure that consecutive chapters are explicitly linked in their summaries.

### World-Building Through Action
- World rules, biology, technology, and social structures must be **revealed through scenes and character experience**, never through exposition dumps.
- The reader should learn about a civilization's rules by seeing someone break them, struggle with them, or rely on them — never by being told about them in an explanatory paragraph.
- Avoid opening a chapter with world-description. Open with action, dialogue, or a character's direct experience, then let the world emerge naturally.

### Context Before Conflict
- Before introducing a conflict or decision, ensure the reader understands **why it matters to this specific character in this specific moment**.
- A dilemma without personal stakes reads like a philosophy essay. Always plant the personal cost first: what does this character stand to lose? What are they afraid of? What do they want that they cannot name?
- Motivation should precede action. If a character makes a surprising choice, the reader must have already seen the seeds of that choice in earlier scenes.

## Quality Bar

- The scaffold must already contain meaningful structure, not just empty folders.
- `story-core.md` should be the contract for what was chosen randomly and why.
- Chapter plans should already carry scene count, block count, required entities, and dialogue modes.
- Avoid bland protagonist/mentor/antagonist defaults unless the user asked for them.
- Prefer unusual but coherent combinations over "most likely" combinations.
- Every chapter plan must include opening-hook and closing-hook fields before prose generation begins.
