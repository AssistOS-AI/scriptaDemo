Specification Driven Book System

How this project is meant to work

This repository is a reusable workspace for managing multiple books through structured specifications, incremental chapter generation, and a shared HTML viewer. Every book lives under docs/BOOKNAME, and the specification layer remains the source of truth.

Source of truth
docs/BOOKNAME/specs stores plans, entities, themes, concepts, and all governing specifications.
Working outputs
docs/BOOKNAME/chapters holds intermediate chapter material derived from the specifications.
Final artifact
docs/BOOKNAME/book.html loads a reusable viewer from docs/preview/preview.js.

Directory model

Each book should have its own isolated specification tree. The manuscript is not treated as a single file. Plans and supporting structures are split into focused Markdown documents so that generation and regeneration can happen with precision.

Core rules

  1. The specification layer defines the intended book structure and constraints.
  2. specs/book-design.md defines the display-facing metadata for the cover and opening pages.
  3. Chapter materials are generated from the specs and remain intermediate assets, not the master source.
  4. The final HTML should assemble content through the viewer library instead of embedding the full book inline.
  5. The build pipeline should be incremental and touch only the outputs affected by a change.
  6. The same workflow must support both literary books and scientific or technical books.

Workflow

The intended lifecycle is controlled, layered, and regenerable. The system should move from idea to plan, from plan to chapter materials, and from chapter materials to a final assembled view.

1

Initialize

Start a new book from a high-level idea and broad instructions. This creates the first specification structure.

2

Refine

Expand the book plan into chapter plans and supporting spec files for entities, themes, concepts, or references.

3

Build

Generate or update the affected chapter Markdown files, refresh the abstract SVG page assets, and rebuild only the sections impacted by recent edits.

4

Assemble

Produce book.html as a thin host page that delegates parsing and presentation to preview.js.

Incremental build behavior

  • If a book-level plan changes, the build skill may need to update multiple chapter plans or metadata outputs.
  • If one chapter specification changes, only the corresponding chapter material and related assembled sections should be regenerated.
  • If an entity file such as a character, location, or concept changes, only the chapters depending on that entity should be rebuilt.
  • If only the viewer changes, chapter Markdown does not need regeneration; only the rendered experience changes.
  • The build logic should track dependencies explicitly so that regeneration is targeted rather than global.

Viewer responsibilities

  • Load the book metadata and chapter Markdown sources.
  • Use the generated cover and opening-page SVG assets as book-facing page art.
  • Parse the Markdown consistently across books.
  • Assemble the final reading order for display.
  • Keep presentation logic reusable and separate from the book content.
  • Allow book.html to stay small, generic, and maintainable.

Skills to use

Four dedicated skills are expected to drive this repository. The names below are the recommended public interface for those skills so that usage stays predictable across books.

Initialization skill

$book-init

Create the first full specification structure for a new book from a concept and broad requirements.

  • Create docs/BOOKNAME/specs and the initial Markdown files.
  • Write a seeded story-core.md with archetypes, relations, story pattern, narrative structure, themes, and pacing defaults.
  • Create richer support groups such as characters, evolution, world building, plot elements, special objects, events, and relationships.
  • Create specs/book-design.md with display metadata, abstract visual direction, and cover/opening copy.
  • Prepare the repository so later chapter building, generation, and aggregation passes can work incrementally.
Planning skill

$chapter-builder

Turn a chapter plan into a draft-ready blueprint with scenes, chapter roles, block cadence, dialogues, and a stricter checklist.

  • Expand ## Scene Blueprint into concrete scene work.
  • Define what each required character is doing in the chapter.
  • Generate a block set and dialogue set that match the chapter rhythm.
  • Strengthen the chapter consistency checklist before prose is drafted.
Generation skill

$chapter-generator

Generate chapter prose from the chapter plan, validate it, and retry until the chapter is structurally consistent.

  • Gather the full chapter packet from Story Core, chapter plan, and support specs.
  • Compose all planned elements into ## Generated Draft.
  • Validate the draft against required characters, places, concepts, events, and objects.
  • Revise and rerun validation instead of accepting broken drafts.
Aggregation skill

$book-build

Aggregate chapter plans into intermediate chapter outputs and assemble the final book view without rebuilding unrelated work.

  • Read the dependency graph for the selected book.
  • Regenerate only the affected chapter files in docs/BOOKNAME/chapters.
  • Generate separate cover-art.svg, cover-page.svg, and opening-page.svg files from specs/book-design.md.
  • Write a global build/consistency-report.json and refresh the final book.html when required.
  • Leave untouched books and unaffected chapters unchanged.
The help page documents the expected contract of these skills. If the repository later introduces different names, keep the behavior equivalent and update this page so the public workflow remains clear.

How to call the skills

Invoke the skills by name and give a precise target book plus the intended scope of work. Keep the request explicit about whether you want initialization, partial regeneration, or a full assembly refresh. 

Use $book-init for a new project.

Example:
Use $book-init to create docs/the-origin-of-patterns.
The book is a scientific work about how recurring structures
emerge in biology, mathematics, and language.
Create a seeded Story Core, a book plan, chapter plans, concept files, and metadata.
Also create book-design metadata for the cover and opening pages.
All specifications must be in Markdown.

Chapter Workflow

Use $chapter-builder for docs/the-origin-of-patterns chapter 03.
Expand scenes, character roles, blocks, and dialogues.

Then use $chapter-generator for the same chapter.
Generate the draft inside the chapter plan source.
Validate and retry until the chapter passes.

Build examples

Use $book-build for incremental aggregation.

Example:
Use $book-build for docs/the-origin-of-patterns.
Only rebuild outputs affected by changes in:
- specs/chapter-plans/03-self-organization.md
- specs/concepts/feedback-loops.md
Update the necessary chapter files, refresh book.html,
and review consistency-report.json if anything fails.

Recommended request pattern

  • Name the book path explicitly, for example docs/my-book.
  • State whether the book is literary, scientific, or hybrid.
  • Provide scope: initialize, build chapter plans, generate chapter prose, or refresh the final assembled view.
  • Mention any special structures that must exist, such as characters, locations, experiments, terminology, relations, or reference concepts.
  • Ask for incremental work unless you truly need a full regeneration.

What not to do

  • Do not treat book.html as the place where the full manuscript is manually maintained.
  • Do not bypass the specification layer by editing generated chapter outputs as if they were the master definition.
  • Do not rebuild every book after every edit.
  • Do not couple viewer logic to one specific title if the same viewer can remain reusable.
